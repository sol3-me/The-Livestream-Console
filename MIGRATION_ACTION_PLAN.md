# The Livestream Console: Architecture Review and Migration Action Plan

## 1) Current architecture (what exists today)

### Application shape

- **Monolithic Node.js Express server** (`server.js`) serves static assets, API endpoints, and page routes in one process.
- **Server-rendered UI with Handlebars** (`express-handlebars`), with route handlers rendering templates in `views/`.
- **Cookie-based auth state**: Google OAuth tokens are stored directly in an HTTP-only cookie named `google_tokens`.
- **No first-party database**: stream data is pulled from YouTube APIs and transformed in memory for view rendering.
- **SASS + Bootstrap styling pipeline**: SCSS is compiled to `public/css/main.css`.

### Request flow

1. User opens `/login` and is redirected to Google consent URL generated by `googleAuth`.
2. Google redirects back to `/auth/redirect` with an auth code.
3. Server exchanges code for tokens and writes tokens to cookie.
4. Protected routes (`/dashboard`, `/streams`, API update/delete routes) gate through `authCheck` middleware.
5. Route/controller calls YouTube Data API through `googleapis` package.

### Core modules

- `server.js`: app wiring, middleware, view engine setup.
- `routes/pageRoutes.js`: page rendering + OAuth callback endpoint.
- `routes/apiRoutes.js`: stream mutation endpoints (stop/edit/delete).
- `controllers/youtubeController.js`: YouTube API integration.
- `middleware/authCheck.js`: cookie presence check + credential injection.
- `models/Stream.js`: maps form payload + existing YouTube stream details to API update body.
- `utils/main.js`: parallel request helper and stream view-model formatting.

## 2) Current technology stack

- **Runtime**: Node.js (project specifies 16.x in engines).
- **Web framework**: Express 4.
- **Templating / SSR**: express-handlebars.
- **Auth integration**: Google OAuth 2.0 via `googleapis` OAuth client.
- **State/session**: browser cookies (no Redis/session store).
- **Frontend styling**: Bootstrap 5 beta + SCSS.
- **Hosting today**: Render free hobby plan (cold starts).

## 3) Observed constraints and pain points

- **Cold start latency dominates UX** on free tier; users abandon before login completes.
- **Token-in-cookie approach** keeps server simple but is fragile for long-lived sessions and scaling.
- **Old dependency baseline** (Express 4 era, Node 16 target, Bootstrap beta), increasing maintenance risk over time.
- **No persistence layer** means limited user/account insights and no easy audit/telemetry.
- **Very low traffic profile** (<< 1 login/month currently; target 3–5/week) means infra simplicity and predictable low fixed cost are more important than throughput.

## 4) SSR options for a rewrite (and fit)

### A) Full SSR Node server (always-on container)

- Framework options: Next.js (Node runtime), Remix, Nuxt (if switching language ecosystem).
- Pros: easy mental model, flexible auth callbacks, rich ecosystem.
- Cons: if kept warm, you pay base monthly compute even with tiny traffic.
- Fit: strong developer ergonomics, less optimal for near-zero baseline cost unless paired with aggressive scaling-to-zero provider.

### B) Hybrid SSR/ISR + edge rendering

- Framework options: Next.js App Router (RSC + route handlers), Astro SSR, SvelteKit.
- Mix static pages (home/about/TOS/privacy) with dynamic authenticated pages.
- Pros: faster perceived performance, lower origin load, can cache anonymous pages heavily.
- Cons: auth flows still need secure server execution path.
- Fit: good balance if expecting occasional bursts and wanting improved startup UX.

### C) Serverless SSR / functions-first

- SSR done in functions, API endpoints in function handlers, static assets on CDN.
- Pros: near-zero idle cost, good for low-frequency usage.
- Cons: cold starts still possible (though often shorter), provider-specific limits and debugging complexity.
- Fit: excellent for current usage profile when tuned to minimize cold path work.

## 5) Low-cost persistence patterns for user accounts

### Option 1: No app DB, delegate user/account to managed auth (recommended for MVP rewrite)

- Use **Auth.js + Google provider** (or provider-native auth like Supabase Auth with Google).
- Keep only minimal user profile in provider-managed store.
- Cost: often free within small MAU limits.
- Benefit: removes most custom auth code and security burden.

### Option 2: Serverless Postgres + ORM for lightweight app metadata

- Providers: Neon/Supabase/Postgres free tier.
- Store users, last-login timestamps, preferences, optional audit events.
- Pair with Drizzle/Prisma.
- Cost: typically free at low usage; low paid tiers if exceeded.

### Option 3: SQLite in container volume (ultra-simple)

- Use SQLite + Prisma/Drizzle with mounted persistent disk.
- Lowest complexity for single-instance deployments.
- Caveat: weak fit for scale-to-zero + multi-instance; still valid for tiny apps.

## 6) Simplifying auth + Google social login

### Current complexity

- Manual OAuth callback handling.
- Manual token cookie write/read.
- Middleware mutates a shared Google auth object per request.

### Modern simplification patterns

1. **Adopt Auth.js (formerly NextAuth.js) with Google-only provider**
   - Handles OAuth flow, CSRF protection, session management, secure cookies.
   - For server frameworks beyond Next.js, use Auth.js adapters.
2. **Use managed auth (Supabase Auth / Clerk / Firebase Auth)**
   - Google login configuration in dashboard.
   - Built-in session rotation and account management.
3. **Use short-lived app session + encrypted server-side token storage**
   - Store refresh/access tokens in DB encrypted at rest.
   - Keep browser cookie as opaque session ID only.

## 7) Three migration stack options (full-stack, Docker-friendly)

## Option A — Next.js 14 + Auth.js + Neon Postgres (recommended default)

- **Frontend/SSR**: Next.js App Router (SSR + RSC + route handlers).
- **Auth**: Auth.js Google provider (single provider).
- **Data**: Neon Postgres + Drizzle ORM.
- **API integration**: Google APIs via server actions/route handlers.
- **Container**: one Docker image running Next standalone output.
- **Why**: excellent ecosystem, fast iteration, strong auth abstractions.

**Rough monthly cost**

- Render Starter web service (always on): ~$7/month.
- Neon free tier DB: $0 (at low usage).
- Total expected: **$0–$10/month** depending on hosting mode.

## Option B — SvelteKit + Lucia/Auth.js + Turso (libSQL)

- **Frontend/SSR**: SvelteKit SSR with progressive enhancement.
- **Auth**: Auth.js compatible flow or Lucia + Google OAuth.
- **Data**: Turso/libSQL (globally replicated SQLite-like service).
- **Container**: Node adapter image in Docker.
- **Why**: very fast runtime, low memory footprint, excellent for tiny services.

**Rough monthly cost**

- Scale-to-zero platform + low DB usage: often **$0–$5/month**.
- Always-on single small instance: **~$5–$10/month**.

## Option C — Astro (static-first) + Cloudflare Workers/Functions + D1/KV + managed auth

- **Frontend/SSR**: Astro static for public pages + SSR islands where needed.
- **Auth/API**: Worker functions for login callbacks and YouTube operations.
- **Data**: D1 (SQLite) and/or KV for session/cache metadata.
- **Container**: optional Docker for local/dev parity; production often non-container edge deploy.
- **Why**: near-zero idle cost and excellent cold-path speed globally.

**Rough monthly cost**

- Low-traffic on free/entry tiers: **$0–$5/month** typically.
- Paid worker safety margin: **~$5/month**.

## 8) Cost control + quick spin-up strategies

1. **Split static and dynamic paths**
   - Deploy marketing/legal pages as static CDN assets.
   - Only wake auth/API runtime for authenticated routes.
2. **Keep runtime minimal**
   - Remove heavy startup work; lazy-initialize Google API clients.
   - Use smaller base image (node:alpine or distroless where suitable).
3. **Use synthetic warm-up sparingly**
   - If provider allows, schedule low-frequency pings only during likely usage windows.
   - Avoid 24/7 pings that defeat free-tier economics.
4. **Cache aggressively where safe**
   - Cache non-sensitive YouTube metadata for short TTL to reduce repeated API calls.
5. **Prefer managed auth + DB free tiers**
   - Offload operational complexity and security maintenance.

## 9) Recommended phased action plan

### Phase 0 — Baseline (1–2 days)

- Add lightweight telemetry: login start -> callback complete timing.
- Measure cold vs warm startup and first-byte time.
- Capture current Render free-tier experience quantitatively.

### Phase 1 — Auth modernization spike (2–4 days)

- Implement Google-only login using Auth.js (or managed auth equivalent) in a POC.
- Remove manual token cookie logic from app code.
- Validate YouTube token refresh lifecycle and revoke handling.

### Phase 2 — SSR rewrite MVP (1–2 weeks)

- Rebuild routes:
  - Public pages static/ISR.
  - `/dashboard` + `/streams` SSR protected routes.
- Port stream operations (stop/edit/delete) behind typed server handlers.
- Add minimal persistence (users + sessions + optional audit log).

### Phase 3 — Cost/perf hardening (2–3 days)

- Optimize container boot time and image size.
- Add caching and short-term response memoization for stream list.
- Configure provider scaling policy and optional working-hours warming.

### Phase 4 — Cutover and monitor (2–3 days)

- Blue/green deploy with rollback plan.
- Compare login completion time before/after.
- Verify monthly cost and adjust plan to keep spend in target band.

## 10) Decision guidance for your specific usage profile

- Given current and near-term volume (3–5 logins/week), prioritize:
  1. **Low idle cost** (scale-to-zero acceptable),
  2. **Fast enough first interaction** (minimize cold path),
  3. **Minimal auth code ownership** (managed/standard auth).
- Practical recommendation:
  - Start with **Option A (Next.js + Auth.js + Neon)** if team is Node/JS-centric and wants easiest hiring/support path.
  - Choose **Option C (Astro/Workers)** if absolute minimum idle cost and quickest wake behavior are top priority and edge platform constraints are acceptable.
